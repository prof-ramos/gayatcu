# GayATCU - Dashboard de Estudos TCU

Este repositório contém o aplicativo GayATCU, um dashboard desenvolvido em Streamlit para o acompanhamento de estudos e revisões referentes ao concurso do TCU, utilizando o Sistema de Repetição Espaçada (SRS).

## Arquitetura e Tecnologias

- **Frontend/Backend**: Streamlit (Python)
- **ORM**: SQLModel (SQLAlchemy + Pydantic)
- **Banco de Dados**: SQLite3 (armazenado em `data/study_tracker.db`)
- **Manipulação de Dados**: Pandas
- **Gráficos**: Plotly
- **Gerenciamento de Dependências**: `uv` (padrão obrigatório conforme regras do ambiente)

## Estrutura de Diretórios e Arquivos Principais

- `app.py`: Arquivo principal (ponto de entrada) que configura a página inicial (Home) e fornece uma visão geral.
- `pages/1_📋_Checklist.py`: Interface para visualização de disciplinas e tópicos (`conteudo.json`) com recursos para marcar o primeiro estudo de cada assunto.
- `pages/2_📅_Revisoes.py`: Motor do SRS. Agrupa tópicos para revisão imediata (atrasados ou de hoje) e exibe um calendário futuro, com base nos intervalos estipulados.
- `pages/3_📊_Estatisticas.py`: Dashboard de acompanhamento com gráficos gerados com Plotly (Métricas de conclusão de disciplina, etc).
- `db.py`: Camada de acesso a dados via SQLModel ORM. Contém modelos (`Topic`, `Progress`, `ReviewLog`) e todas as funções de consulta/mutação.
- `session.py`: Gerenciamento do ciclo de vida e estado (singleton) do engine SQLAlchemy no Streamlit, garantindo concorrência segura na sessão do usuário.
- `utils.py`: Funções utilitárias e regras (cálculo de datas das revisões baseada em deltas, load_content, formatações).
- `components/charts.py`: Componentes reutilizáveis de gráficos Plotly (donut, bar, line, pie).
- `tests/conftest.py`: Fixtures compartilhadas (engine SQLite in-memory com StaticPool).
- `tests/test_db.py`: Testes unitários da camada de dados (import, marcação, revisões, estatísticas, exportação).
- `tests/test_utils.py`: Testes unitários de funções utilitárias (SRS, formatação, validação JSON).

## Schema do Banco de Dados

### Tabela `topics`
| Campo    | Tipo   | Descrição                    |
|----------|--------|------------------------------|
| id       | INT PK | Identificador auto-increment |
| codigo   | TEXT   | Código do tópico (indexado)  |
| secao    | TEXT   | Seção/disciplina (indexada)  |
| subsecao | TEXT   | Subseção                     |
| titulo   | TEXT   | Título do tópico             |

**Constraint**: UNIQUE(codigo, secao, subsecao)

### Tabela `progress`
| Campo            | Tipo      | Descrição                           |
|------------------|-----------|-------------------------------------|
| topic_id         | INT PK/FK | FK → topics.id                      |
| completed_at     | TEXT/NULL  | ISO datetime da conclusão           |
| last_reviewed_at | TEXT/NULL  | ISO datetime da última revisão      |
| review_count     | INT       | Contagem de revisões realizadas     |
| next_review_date | TEXT/NULL  | Data (YYYY-MM-DD) da próxima revisão|

### Tabela `review_log`
| Campo         | Tipo      | Descrição                         |
|---------------|-----------|-----------------------------------|
| id            | INT PK    | Identificador auto-increment      |
| topic_id      | INT FK    | FK → topics.id                    |
| reviewed_at   | TEXT      | ISO datetime da revisão           |
| interval_days | INT       | Intervalo em dias desta revisão   |

**Formato de datas**: ISO 8601 (`datetime.isoformat()`) para timestamps, `YYYY-MM-DD` para datas puras.

## Regras de Negócio do Sistema de Repetição Espaçada (SRS)

1. **Início do Ciclo**: Quando um tópico é concluído no checklist, ele é inserido no banco de dados e ganha status de Nível 0 (próxima revisão em 24h).
2. **Intervalos**:
   - Revisão Nível 0: + 1 dia (24 horas)
   - Revisão Nível 1: + 7 dias
   - Revisão Nível 2: + 15 dias
   - Revisão Nível 3+: + 30 dias (intervalo fixo de consolidação)
3. **Penalidades**:
   - **Revisão incorreta**: O ciclo retorna ao Nível 0 (próxima revisão em 24h), sem exceções.
   - **Revisão perdida** (timeout — tópico passou da data agendada sem ser revisado): Tratada da mesma forma que revisão incorreta — retorna ao Nível 0. A detecção ocorre ao carregar a página de revisões (`pages/2_📅_Revisoes.py`), que compara `next_review_date` com a data atual.
   - Não existe fluxo de "acerto parcial" na implementação atual.

## Diretrizes de Desenvolvimento (LLM Context)

- **Idioma de Comunicação e Código**: O idioma padrão para comentários, documentações e comunicação nos testes/features DEVE ser o **Português Brasileiro (PT-BR)**.
- **Comandos de Terminal**: Utilize o gerenciador **`uv`** obrigatoriamente (`uv run streamlit run app.py`).
- **Testes**: `uv run pytest tests/ -v` (40 testes cobrindo db.py e utils.py).
- **Padrões de Qualidade**:
  - Toda interação com banco DEVE usar as funções ORM centralizadas no `db.py` — **nunca** SQL raw nas pages.
  - Componentes do Streamlit que precisam do banco interagem via `get_db()` de `session.py` (retorna um `engine` SQLAlchemy).
  - Lógica do banco de dados NÃO deve vazar para as páginas frontend (`pages/`).
  - Tratamento de erros: capture exceções específicas — evite `except Exception` genérico.
  - Logging: use o módulo `logging` (não `print`), incluindo contexto nas mensagens.
  - Formatação: Ruff como linter/formatter (configurado em `pyproject.toml`).
